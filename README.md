# NyamkaBot

Discord-бот для верификации участников через анкеты, с модерацией заявок, апелляциями и публикацией готовых embed-сообщений (`fluffycore`, `info1`, `info2`, `navigation`, `rules1`, `rules2`, `terms`). Дополнительно автоматически выдаёт роль за тег сервера (Server Tag). Команды — слеш-команды (`/верификация`, `/апелляция` — owner; `/запостить`, `/амнистии`, `/выдатьчсп`, `/снятьчсп` — ststaff; `/анкеты`, `/тег` — staff), данные хранятся во внешней **MySQL/MariaDB** (через `mysql2`). Бот **мультисерверный**: один процесс обслуживает все серверы, на которых он есть, а роли и каналы настраиваются для каждого сервера отдельно.

> Краткий обзор всех возможностей — в [`docs/features.md`](docs/features.md). Справочник по правам доступа — в [`docs/permissions.md`](docs/permissions.md). Подробный разбор всех сценариев верификации и апелляций — в [`docs/verification-and-appeals.md`](docs/verification-and-appeals.md).

> Бот использует привилегированный intent **Server Members Intent** (`GuildMembers`) — включите его в Discord Developer Portal → Bot → Privileged Gateway Intents, иначе проверки ролей и синхронизация тега работать не будут.
>
> Для определения **способа вступления** (по какому инвайту зашёл участник) боту нужен intent `GuildInvites` (включается в коде) и право **«Управление сервером»** на сервере — без него поле «Способ вступления» в анкете всегда будет «Неизвестно».
>
> Для **автоотключения с голосовых каналов** участников в чёрном списке боту нужен intent `GuildVoiceStates` (включается в коде).

## 2. Быстрый старт с Docker

### 2.1. Предварительные требования

- Установленный Docker (вместе с Docker Compose).
- Доступная база **MySQL/MariaDB** (на отдельном хосте или в соседнем контейнере) — бот хранит в ней все данные.

Если Docker ещё не установлен, поставь его одной командой:

```bash
curl -fsSL https://get.docker.com -o get-docker.sh
sh get-docker.sh
```

### 2.2. Установка и запуск

1. **Клонируй репозиторий:**

    ```bash
    git clone https://github.com/Alexnitro777/NyamkaBot.git
    cd NyamkaBot
    ```

2. **Создай `docker-compose.yml`** из шаблона и задай свои `image`, `container_name` и переменные окружения:

    ```bash
    cp docker-compose.example.yml docker-compose.yml
    ```

    В блоке `environment` укажи подключение к БД (`DB_HOST`, `DB_USER`, `DB_PASSWORD`, `DB_NAME`). При первом запуске также задай `BOT_TOKEN` и `CLIENT_ID` — бот запишет их в таблицу `app_config` и дальше будет брать оттуда (после старта эти две переменные можно убрать). Либо внеси `token` и `clientId` напрямую в таблицу `app_config` (key/value) и не указывай их в env. Вся остальная конфигурация хранится в БД.

3. **Запусти бота:**

    ```bash
    docker compose up -d --build
    ```

    При старте контейнер создаёт таблицы в БД, читает `token`/`clientId` из `app_config` (или из env при первом запуске) и автоматически регистрирует слеш-команды на всех серверах, где он есть, — отдельного шага деплоя команд нет.

4. **Настрой серверы.** Конфигурации в файле нет — настройки каждого сервера лежат в таблице `guild_settings` (по строке на каждый ключ). Шаги ниже повтори для **каждого** сервера, где работает бот.

    **4.1. Включи «Режим разработчика» в Discord.** Настройки пользователя → **Расширенные** → включи **«Режим разработчика»**. После этого в контекстном меню (правый клик) появится пункт **«Копировать ID»**.

    **4.2. Скопируй нужные ID.** Все значения в конфиге — это числовые Discord-ID (snowflake). Получаешь их правым кликом → **«Копировать ID»**:

    - **ID сервера** — правый клик по иконке сервера → «Копировать ID сервера».
    - **ID роли** — Настройки сервера → **Роли** → правый клик по роли → «Копировать ID».
    - **ID канала** — правый клик по текстовому каналу → «Копировать ID».
    - **ID категории** — правый клик по **категории** (заголовку группы каналов) → «Копировать ID».

    **4.3. Подготовь роли и каналы** (создай заранее, если их ещё нет): роль верифицированных и роль ЧС; роли админов и модерации; канал очереди анкет и канал очереди апелляций; категорию для приватных каналов-вопросов.

    > Роль бота должна стоять **выше** в иерархии всех ролей, которые он выдаёт/снимает (verified, ЧС, роль за тег) — иначе Discord не даст ими управлять.

    **4.4. Вставь строки в `guild_settings`** через SQL или GUI (phpMyAdmin / Adminer / DBeaver). Ключи — с точками; `roles.staff` и `roles.ststaff` — это списки ID ролей через запятую. Замени плейсхолдеры на свои ID:

    ```sql
    INSERT INTO guild_settings (guildId, `key`, value) VALUES
    -- обязательные:
    ('ID_СЕРВЕРА', 'roles.verified',        'ID_РОЛИ'),
    ('ID_СЕРВЕРА', 'roles.blacklist',       'ID_РОЛИ'),
    ('ID_СЕРВЕРА', 'roles.staff',           'ID_РОЛИ1,ID_РОЛИ2'),
    ('ID_СЕРВЕРА', 'roles.ststaff',         'ID_РОЛИ1,ID_РОЛИ2'),
    ('ID_СЕРВЕРА', 'channels.review',       'ID_КАНАЛА'),
    ('ID_СЕРВЕРА', 'channels.appealReview', 'ID_КАНАЛА'),
    ('ID_СЕРВЕРА', 'questionCategoryId',    'ID_КАТЕГОРИИ'),
    -- необязательные (можно пропустить):
    ('ID_СЕРВЕРА', 'roles.blacklistSoft',   'ID_РОЛИ1,ID_РОЛИ2'),
    ('ID_СЕРВЕРА', 'roles.roleTag',         'ID_РОЛИ'),
    ('ID_СЕРВЕРА', 'channels.welcome',      'ID_КАНАЛА'),
    ('ID_СЕРВЕРА', 'channels.decisions',    'ID_КАНАЛА'),
    ('ID_СЕРВЕРА', 'channels.appeal',       'ID_КАНАЛА'),
    ('ID_СЕРВЕРА', 'channels.tagLog',       'ID_КАНАЛА'),
    ('ID_СЕРВЕРА', 'channels.blacklistLog', 'ID_КАНАЛА');
    ```

    Назначение ключей:

    | Ключ | Обяз. | Что задаёт |
    |---|---|---|
    | `roles.verified` | да | Роль, выдаваемая при принятии анкеты |
    | `roles.blacklist` | да | Роль ЧС (выдаётся при заносе в чёрный список) |
    | `roles.blacklistSoft` | нет | Доп. роли ЧС через запятую — участники могут подавать апелляции, но не отключаются с голосовых каналов |
    | `roles.staff` | да | Роли staff через запятую — только анкеты: `/анкеты`, кнопки решения по анкетам, `/тег` |
    | `roles.ststaff` | да | Роли ststaff через запятую — анкеты + апелляции + `/запостить` + `/выдатьчсп` / `/снятьчсп` |
    | `channels.review` | да | Канал очереди анкет на проверку |
    | `channels.appealReview` | да | Канал очереди апелляций на проверку |
    | `questionCategoryId` | да | Категория, в которой создаются приватные каналы-вопросы |
    | `roles.roleTag` | нет | Роль, выдаваемая за тег сервера (Server Tag) |
    | `channels.welcome` | нет | Канал приветствия принятых участников |
    | `channels.decisions` | нет | Канал-лог решений по анкетам и апелляциям |
    | `channels.appeal` | нет | Канал с кнопкой подачи апелляции |
    | `channels.tagLog` | нет | Лог выдачи/снятия роли за тег |
    | `channels.blacklistLog` | нет | Лог команд `/выдатьчсп` и `/снятьчсп` |

    Необязательные ключи (после `-- необязательные`) можно не вставлять — без них соответствующие функции (роль за тег, приветствие, логи) просто отключены. Если же пропустить хотя бы один **обязательный** ключ, бот на этом сервере ответит «⚠️ Бот не настроен на этом сервере».

    **Уровни доступа** (строгий режим — доступ дают только эти роли и owner-ID, право Discord «Администратор» само по себе ничего не даёт):

    - **staff** (`roles.staff`) — только анкеты: `/анкеты`, кнопки решения по анкетам, `/тег`.
    - **ststaff** (`roles.ststaff`) — всё, что staff, плюс `/запостить`, апелляции (`/амнистии`, кнопки решения по апелляциям) и `/выдатьчсп` / `/снятьчсп`.
    - **owner** — **зашит в коде**, не в БД: массив `OWNER_IDS` в `src/permissions.ts`. Доступны **все** команды, включая `/верификация` и `/апелляция` (они owner-only на любом сервере). Впиши сюда свой Discord user ID и пересобери образ (`docker compose up -d --build`).

5. **Перечитай настройки** — после добавления строк перезапусти бота, чтобы он сбросил кэш конфигов серверов:

    ```bash
    docker compose restart
    ```

## 3. Управление контейнером

- **Логи в реальном времени** — `docker compose logs -f`
- **Остановить бота** — `docker compose stop`
- **Запустить ранее собранный контейнер** — `docker compose start`
- **Обновить конфиг сервера** — измени строки в таблице `guild_settings`, затем `docker compose restart` (рестарт сбрасывает кэш настроек серверов).
- **Перезапустить** (после правки переменных окружения в `docker-compose.yml` или изменений в БД) — `docker compose restart`
- **Пересобрать** (после изменений в коде или добавления команд) — `docker compose up -d --build`
- **Остановить и удалить контейнер** — `docker compose down`

> Конфиг-файлов на диске нет: подключение к БД задаётся переменными `DB_*` в `docker-compose.yml`, а `token`/`clientId` и настройки серверов лежат в самой MariaDB (таблицы `app_config` и `guild_settings`). Все данные живут во внешней БД и не зависят от пересборки образа. Перезапуск нужен после изменения переменных окружения или строк конфигурации в БД, а пересборка — при изменении кода или добавлении новых команд. Только `docker-compose.yml` не в git — это твоя локальная копия шаблона.
