# Верификация и апелляции — полный разбор сценариев

Документ описывает **все** варианты прохождения верификации и подачи апелляции в FemVerify-BOT —
со стороны **участника** и со стороны **модератора**. Основано на коде проекта
(`src/buttons`, `src/modals`, `src/commands`, `src/storage.ts`, `src/ui.ts`, `src/inviteTracker.ts`,
`src/*Cleanup.ts`, `src/leaveCleanup.ts`).

---

## 0. Глоссарий статусов

В базе (MySQL/MariaDB) у каждого пользователя в пределах одного сервера хранится не более одной
заявки и одной апелляции (составной ключ — `(guildId, userId)`, новая запись перезаписывает старую
через `ON DUPLICATE KEY UPDATE`).

**Статусы анкеты** (`ApplicationStatus`):

| Статус | Что значит | Кто ставит |
|---|---|---|
| `pending` | На рассмотрении | Участник (отправка анкеты) |
| `approved` | Одобрена, выдана роль verified | Модератор (кнопка «Принять») |
| `rejected` | Отклонена, можно переподать | Модератор (кнопка «Отклонить» + причина) |
| `blacklisted` | В чёрном списке, доступ только к апелляции | Модератор (кнопка «ЧС» + причина, либо команда `/выдатьчсп`) |
| `left` | Участник покинул сервер с неразобранной заявкой | Авто (событие выхода) |
| `expired` | Заявка авто-закрыта по истечении 48 ч | Авто (sweep `applicationCleanup`) |

**Статусы апелляции** (`AppealStatus`):

| Статус | Что значит | Кто ставит |
|---|---|---|
| `pending` | На рассмотрении | Участник (отправка апелляции) |
| `amnestied` | Амнистия принята, ЧС снят | Модератор (кнопка «Принять амнистию») |
| `denied` | В амнистии отказано, ЧС сохраняется, действует кулдаун 48 ч | Модератор (кнопка «Отказать в амнистии») |
| `left` | Участник покинул сервер с неразобранной апелляцией | Авто (событие выхода) |

**Тайминги (константы кода):**

- TTL анкеты: **48 ч** (`APPLICATION_TTL_MS = 2 * 24 ч`) → после этого `expired`.
- TTL канала-вопроса по **анкете**: **48 ч / 2 дня** (`QUESTION_TTL_MS`) → канал авто-удаляется (совпадает с TTL анкеты). Каналы-вопросы **апелляций** по TTL не удаляются — только при решении апелляции или выходе участника.
- Кулдаун повторной апелляции после отказа: **48 ч** (`DENY_COOLDOWN_MS`).
- Период проверок (sweep): каждые **5 мин** (или TTL/4, что меньше).

# ЧАСТЬ I. ВЕРИФИКАЦИЯ

## I-A. Со стороны УЧАСТНИКА

### 1. Точка входа

Администратор командой `/верификация` размещает в канале embed «➕ Добро пожаловать!»
с зелёной кнопкой **«Пройти верификацию»** (`verify:start`). Участник нажимает её.

### 2. Проверки ДО открытия анкеты (`verifyStart.ts`)

При нажатии кнопки бот по очереди проверяет (всё — эфемерные ответы, видны только нажавшему):

| # | Условие | Реакция бота | Может ли продолжить |
|---|---|---|---|
| 2.1 | У участника есть роль **ЧС** (`roles.blacklist`) | «Вы находитесь в чёрном списке. Используйте канал апелляции.» | ❌ нет |
| 2.2 | Существует заявка со статусом `pending` | «Ваша заявка уже на рассмотрении.» | ❌ нет |
| 2.3 | Заявка `approved` **И** уже есть роль verified | «Вы уже верифицированы.» | ❌ нет |
| 2.4 | Ни одно из условий выше | Открывается модальное окно анкеты | ✅ да |

> Важный нюанс п. 2.3: блокировка срабатывает, только если статус `approved` **и** роль
> реально присутствует. Если роль сняли вручную (а статус остался `approved`) — участник
> сможет подать заявку заново.
>
> Статусы `rejected`, `expired`, `left` **не блокируют** повторную подачу — это и есть
> механизм «переподать заявку».

### 3. Заполнение анкеты (модалка `verify:submit`)

Анкета содержит **5 полей** (`verifyQuestions`; Discord допускает максимум 5):

1. **Откуда узнал о сервере?** — абзац, обязательно, до 200 символов.
2. **Что ожидаешь от сервера?** — абзац, обязательно, до 200 символов.
3. **Сколько вам лет?** — короткое поле, обязательно, до 2 символов.
4. **Твоё отношение к фурри/фембой сообществу?** — абзац, обязательно, до 300 символов.
5. **Правила прочитаны и приняты?** — абзац, обязательно, до 30 символов.

### 4. Отправка анкеты (`verifySubmit.ts`)

| Сценарий | Что происходит |
|---|---|
| **Успех** | Заявка `pending` сохраняется в БД; embed заявки публикуется в канал модерации (`channels.review`) с 4 кнопками; участнику: «✅ Анкета отправлена. Ожидайте решения модерации.» |
| **Канал модерации недоступен** | «❌ Не удалось отправить заявку: канал модерации недоступен…» — заявка **не сохраняется** |
| **Не удалось отправить сообщение в канал** | «❌ Не удалось отправить заявку модерации. Попробуйте позже…» — заявка **не сохраняется** |

> Пустые ответы (если поле не получено) подставляются как пустая строка и в embed отображаются как «—».
> Длинные ответы обрезаются до 1000 символов.

### 5. Возможные исходы для участника после отправки

#### 5.1. ✅ Заявку ОДОБРИЛИ
- Участнику в ЛС: embed «✅ Заявка одобрена — Добро пожаловать на сервер!».
- Выдаётся роль verified.
- В канал `welcome` (если настроен) — приветственный embed с упоминанием участника.
- Если ЛС закрыты — роль всё равно выдаётся, участник просто не получит сообщение.

#### 5.2. ❌ Заявку ОТКЛОНИЛИ
- В ЛС: «❌ Заявка отклонена — Причина: `<текст модератора>`. Вы можете подать новую заявку.»
- Роль не выдаётся. Можно **переподать** анкету (статус `rejected` не блокирует).

#### 5.3. 🚫 Отправили в ЧС
- В ЛС: «🚫 Вы добавлены в чёрный список — Причина: `<текст>`. Можете подать апелляцию в `<канал>`.»
- **Снимаются все роли** участника, выдаётся роль ЧС. Повторная верификация заблокирована
  (см. п. 2.1) — остаётся только апелляция.

#### 5.4. ❓ Модератор ЗАДАЛ ВОПРОС
- Создаётся приватный текстовый канал `вопрос-<имя>` в категории `questionCategoryId`.
- Доступ виден только участнику и ролям `roles.staff` + `roles.ststaff`.
- Внутри — embed «Уточнение по заявке» с кнопкой-ссылкой на анкету и кнопкой «Закрыть канал».
- Участник отвечает в канале. Заявка остаётся `pending` до финального решения.
- Канал авто-удаляется через **2 дня (48 ч)** — вместе с автозакрытием самой анкеты (TTL анкеты тоже 2 дня).
- При любом финальном решении (принять/отклонить/ЧС/expired) канал-вопрос удаляется автоматически.

#### 5.5. ⌛ Заявка АВТО-ЗАКРЫТА (expired)
- Если за **48 ч** модерация не приняла решение — sweep ставит статус `expired`.
- В ЛС: «⌛ Заявка закрыта — Переподайте заявку на верификацию.»
- Сообщение в канале модерации помечается «Закрыто автоматически».
- Канал-вопрос (если был) удаляется. Можно подать анкету заново.

#### 5.6. 🚪 Участник ПОКИНУЛ сервер (left)
- Если у участника была `pending`-заявка и он вышел — статус `left`.
- Сообщение в канале модерации помечается кнопкой «Покинул сервер».
- Канал-вопрос (если был) удаляется.

### 6. Сводка ограничений участника при верификации

- Одна активная (`pending`) заявка за раз.
- В ЧС — верификация недоступна (только апелляция).
- После `rejected` / `expired` / `left` — можно подавать заново без кулдауна.
- ЛС стоит держать открытыми, иначе результат не придёт (роль/ЧС всё равно применятся).

## I-B. Со стороны МОДЕРАТОРА

### 1. Где появляются заявки

Каждая новая анкета приходит в канал `channels.review` как embed. В шапке embed выводятся
**участник**, **дата создания аккаунта**, **дата захода на сервер** и **способ вступления** —
реальный инвайт, по которому участник зашёл (упоминание пригласившего, vanity-ссылка или
«Неизвестно»; определяется трекером инвайтов `inviteTracker.ts` при входе). Ниже идут все 5
ответов анкеты, а под embed — строка из **4 кнопок**. Доступ к кнопкам проверяется
`hasButtonAccess(..., 'staff')`: нужна роль `roles.staff`/`roles.ststaff` или owner-ID. Иначе —
«Недостаточно прав.».

### 2. Четыре решения по заявке (`review.ts`)

| Кнопка | customId | Действие | Требует причину |
|---|---|---|---|
| ✅ **Принять** | `review:approve:<id>` | Выдаёт роль verified, шлёт ЛС, постит welcome | Нет |
| ❌ **Отклонить** | `review:reject:<id>` | Открывает модалку причины → статус `rejected` | Да (до 1000 симв.) |
| ❓ **Задать вопрос** | `review:question:<id>` | Создаёт приватный канал-вопрос | Нет |
| 🚫 **ЧС** | `review:blacklist:<id>` | Открывает модалку причины → статус `blacklisted`, снимает все роли + выдаёт роль ЧС | Да (до 1000 симв.) |

### 3. Детали по каждому решению

#### 3.1. ✅ Принять
- Атомарный «claim»: `claimApplication(... 'approved')` срабатывает только если статус ещё `pending`.
  Если двое модераторов нажали одновременно — второй получит «Заявка уже обработана (…)».
- Если участник вышел с сервера — «Пользователь покинул сервер.».
- **Ошибка выдачи роли** (роль бота ниже выдаваемой): статус откатывается в `pending`,
  модератору — «❌ Не удалось выдать роль — проверьте, что роль бота выше выдаваемой…».
- После успеха: embed заявки перекрашивается в зелёный с пометкой модератора, кнопки заменяются
  на неактивную «Анкета обработана», в `channels.decisions` постится решение, канал-вопрос удаляется.
- Если ЛС участника закрыты — «⚠️ Роль выдана, но отправить ЛС не удалось».

#### 3.2. ❌ Отклонить (`reviewReason.ts`)
- Сначала проверка: если заявка уже не `pending` — «Заявка уже обработана (…)», модалка не откроется.
- Модератор вводит причину → статус `rejected`, причина сохраняется.
- Участнику ЛС с причиной, embed помечается «Отклонено» (красный), решение в `decisions`,
  канал-вопрос удаляется.

#### 3.3. ❓ Задать вопрос
- Если канал-вопрос уже существует — «Канал с вопросом уже существует: <#…>» (повторно не создаётся).
- Если участник вышел — «Пользователь покинул сервер.».
- Иначе создаётся приватный канал с правами для участника и ролей `roles.staff` + `roles.ststaff`,
  ID канала сохраняется в заявке. Заявка остаётся `pending` — решение принимается позже теми же
  4 кнопками (на сообщении модерации кнопка «Задать вопрос» меняется на ссылку «Перейти к вопросу»).
- Кнопка **«Закрыть канал»** внутри канала-вопроса доступна с уровня **staff** и выше
  (`hasButtonAccess(..., 'staff')`).

#### 3.4. 🚫 ЧС (`reviewReason.ts`)
- Аналогично отклонению, но статус `blacklisted`; затем `blacklistMemberRoles(member)` (`roles.ts`)
  **снимает все роли** участника и выдаёт роль `roles.blacklist` одним `roles.set`.
- Сохраняются роли, которые бот не может трогать: управляемые (бустер, интеграции) и роли выше
  бота по иерархии.
- Список снятых ролей сохраняется в анкете (`removedRoles`) — он используется для возврата ролей при амнистии.
- Если обновить роли не удалось (иерархия) — «⚠️ Не удалось обновить роли (ЧС)…», статус всё равно `blacklisted`.
- В ЛС участнику — текст с причиной и указанием канала апелляций.

### 4. Конкурентность и идемпотентность

- Все финальные решения проходят через атомарный `claimApplication` (UPDATE … WHERE status='pending').
  Это исключает двойную обработку — выигрывает первый нажавший, остальным сообщается актуальный статус.
- Кнопки «Отклонить»/«ЧС» дополнительно проверяют `pending` ещё до показа модалки.

### 5. Обзор очереди

- `/анкеты` (доступ `mod`) — список всех `pending`-анкет: номер заявки, упоминание, тег, время
  подачи `<t:…:R>` и ссылка «перейти» на сообщение модерации. Если очередь пуста —
  «Непринятых анкет нет. 🎉». Ответ эфемерный.

### 6. Что модератор видит при авто-событиях

- **expired**: сообщение перекрашивается в серый, кнопка «Закрыто автоматически», запись в `decisions`.
- **left**: на сообщении заявки появляется неактивная кнопка «Покинул сервер».

### 7. Команда `/выдатьчсп` — занесение в ЧС напрямую (`commands/chsp.ts`)

Заносит в ЧС **любого** участника сервера, без анкеты. Доступ `mod` (видна при праве
**«Управление никами»**). Команда открывает модалку `chsp:reason:<userId>` (обработчик
`modals/chspReason.ts`) с обязательным полем причины (до 1000 симв.). Ответ эфемерный.

При подтверждении модалки:
- `blacklistMemberRoles(member)` (`roles.ts`) **снимает все роли** и выдаёт роль `roles.blacklist`
  одним `roles.set`; управляемые роли (бустер, интеграции) и роли выше бота по иерархии сохраняются.
  Снятые роли запоминаются в `removedRoles` анкеты и возвращаются при амнистии или через `/снятьчсп`.
- Статус анкеты участника выставляется в `blacklisted` с причиной (`updateApplication`, либо
  `saveApplication` с пустыми ответами, если анкеты не было). Это форс-апсерт, а не атомарный
  `claimApplication`: команда — принудительное действие над произвольным участником, статус может
  быть любым (или анкеты нет вовсе).
- Участнику ЛС с причиной и указанием канала апелляций; решение «ЧС» пишется в `channels.blacklistLog`.
- Если снять роли не удалось (иерархия) — в ответе «⚠️ Не удалось снять все роли…», но ЧС всё равно проставлен.

Дальше работает штатная апелляция: `appealSubmit.ts` видит роль ЧС и подставляет причину из
анкеты `blacklisted`. Если у участника была активная (`pending`) карточка-анкета, она станет
`blacklisted`, но само сообщение в `channels.review` не гасится — его кнопки «Принять/Отклонить»
из-за гарда `claimApplication` лишь сообщат «уже обработана».

### 8. Команда `/снятьчсп` — снятие с ЧС напрямую (`commands/unchsp.ts`)

Снимает с ЧС **любого** участника сервера. Доступ `mod` (видна при праве **«Управление никами»**).
Команда открывает модалку `unchsp:reason:<userId>` (обработчик `modals/unchspReason.ts`) с
обязательным полем причины (до 1000 симв.). Ответ эфемерный.

При подтверждении модалки:
- Снимается роль `roles.blacklist` (`member.roles.remove`).
- Если в анкете сохранены `removedRoles`, они возвращаются через `restoreMemberRoles` (`roles.ts`),
  затем `removedRoles` очищается — ровно как при принятии амнистии.
- **Статус анкеты не меняется** (остаётся прежним, обычно `blacklisted`): это, как и амнистия,
  лишь снятие роли и возврат ролей, без атомарного `claim*`.
- Участнику ЛС о снятии ЧС; решение «Снят с ЧС» пишется в `channels.blacklistLog`.
- Если снять роль ЧС или вернуть роли не удалось (иерархия) — в ответе соответствующее
  предупреждение «⚠️ …».

# ЧАСТЬ II. АПЕЛЛЯЦИИ

## II-A. Со стороны УЧАСТНИКА

### 1. Точка входа

Администратор командой `/апелляция` размещает embed «🔓 Подать апелляцию» с кнопкой
**«Подать апелляцию»** (`appeal:start`). Кнопка предназначена для участников в ЧС.

### 2. Проверки ДО открытия формы (`appealStart.ts`)

| # | Условие | Реакция бота | Продолжит |
|---|---|---|---|
| 2.1 | У участника **нет** роли ЧС | «Апелляция доступна только участникам в чёрном списке.» | ❌ нет |
| 2.2 | Есть апелляция `pending` | «Ваша апелляция уже на рассмотрении.» | ❌ нет |
| 2.3 | Прошлая апелляция `denied` и не прошло **48 ч** | «⛔ Вашу прошлую апелляцию отклонили. Новую можно подать `<t:…:R>`.» | ❌ нет |
| 2.4 | Ничего из вышеперечисленного | Открывается форма апелляции | ✅ да |

> После истечения 48 ч с момента отказа (`resolvedAt + DENY_COOLDOWN_MS`) можно подать снова.
> Те же проверки повторяются и при отправке формы (`appealSubmit.ts`).

### 3. Форма апелляции (модалка `appeal:submit`)

Одно поле (`appealQuestions`):
- **Текст апелляции** — абзац, обязательно, от 20 до 150 символов.
  Подсказка: «Опиши спокойно: за что, как ты понял ситуацию, и почему стоит дать второй шанс.»

### 4. Отправка апелляции (`appealSubmit.ts`)

| Сценарий | Что происходит |
|---|---|
| **Успех** | Апелляция `pending` сохраняется; embed «Апелляция №N» публикуется в `channels.appealReview` с 3 кнопками; участнику: «✅ Апелляция отправлена. Ожидайте решения модерации.» |
| **Канал апелляций недоступен** | «❌ Не удалось отправить апелляцию: канал модерации недоступен…» — не сохраняется |
| **Не удалось отправить сообщение** | «❌ Не удалось отправить апелляцию модерации. Попробуйте позже…» — не сохраняется |

> Если у участника есть анкета в статусе `blacklisted`, её причина ЧС подставляется в embed
> апелляции отдельным полем «Причина ЧС».

### 5. Возможные исходы для участника

#### 5.1. ✅ АМНИСТИЯ принята
- В ЛС: «✅ Амнистия принята — С вас снят чёрный список. Вы можете снова пройти верификацию.»
- Роль ЧС снимается. Статус апелляции `amnestied`. Участник снова может проходить верификацию.
- **Роли, снятые при занесении в ЧС, возвращаются** (те, что ещё существуют и доступны боту по иерархии).

#### 5.2. ❌ В амнистии ОТКАЗАНО
- В ЛС: «❌ В амнистии отказано — Ваша апелляция отклонена. ЧС сохраняется. Вы сможете подать
  новую апелляцию `<t:…:R>`.»
- Статус `denied`, фиксируется `resolvedAt`. Включается **кулдаун 48 ч** на новую апелляцию.

#### 5.3. ❓ Модератор ЗАДАЛ ВОПРОС
- Создаётся приватный текстовый канал `вопрос-<имя>` в категории `questionCategoryId`.
- Доступ виден только участнику и ролям `roles.ststaff` (staff каналы апелляций не видит).
- Внутри — embed «Уточнение по апелляции» с кнопкой-ссылкой на апелляцию и кнопкой «Закрыть вопрос».
- Участник отвечает в канале. Апелляция остаётся `pending` до финального решения.
- Канал **по таймеру не удаляется** (в отличие от каналов-вопросов по анкете): пока участник на сервере, он живёт.
- Канал-вопрос удаляется автоматически только при финальном решении (амнистия/отказ) **или при выходе участника** с сервера (`leaveCleanup.ts`).

#### 5.4. 🚪 Участник ПОКИНУЛ сервер (left)
- `pending`-апелляция при выходе помечается `left`, сообщение в канале — кнопкой «Покинул сервер».
- Канал-вопрос (если был) удаляется.

### 6. Сводка ограничений участника при апелляции

- Доступна только при наличии роли ЧС.
- Одна активная (`pending`) апелляция за раз.
- После отказа — кулдаун 48 ч.
- Текст: 20–150 символов; оскорбления/угрозы/шантаж по правилам ведут к отказу (см. embed `/апелляция`).
- ЛС держать открытыми, иначе результат не придёт (роль ЧС всё равно снимется при амнистии).

## II-B. Со стороны МОДЕРАТОРА

### 1. Где появляются апелляции

Каждая апелляция приходит в `channels.appealReview` как embed «Апелляция №N»
(участник, дата создания аккаунта, причина ЧС — если есть, текст, ID) с **3 кнопками**.
Доступ к кнопкам — уровень **ststaff** и выше (`hasButtonAccess(..., 'ststaff')`): роль
`roles.ststaff` или owner-ID. Staff (только анкеты) апелляции не обрабатывает.

### 2. Три действия по апелляции (`appealReview.ts`)

| Кнопка | customId | Действие | Требует причину |
|---|---|---|---|
| ✅ **Принять амнистию** | `appeal:amnesty:<id>` | Снимает роль ЧС, статус `amnestied`, ЛС участнику | Нет |
| ❌ **Отказать в амнистии** | `appeal:deny:<id>` | Статус `denied`, ЛС участнику, запускает кулдаун 48 ч | Нет |
| ❓ **Задать вопрос** | `appeal:question:<id>` | Создаёт приватный канал-вопрос | Нет |

### 3. Детали

#### 3.1. ✅ Принять амнистию
- Атомарный `claimAppeal(... 'amnestied')` — только если апелляция ещё `pending`,
  иначе «Апелляция уже обработана (…)».
- Снимается роль ЧС. Если снять не удалось (иерархия) — «⚠️ Не удалось снять роль ЧС…»,
  но статус всё равно `amnestied`.
- **Возврат ролей**: добавляются роли из `application.removedRoles`, которые ещё существуют и ниже бота
  по иерархии; затем список очищается. При неудаче — предупреждение «Не удалось вернуть часть ролей».
- Если участник вышел — роль не снимается, но статус проставляется; ЛС не доставится.
- Embed перекрашивается в зелёный с пометкой модератора, кнопки → неактивная «Апелляция обработана»,
  запись в `channels.decisions`, канал-вопрос (если был) удаляется.

#### 3.2. ❌ Отказать в амнистии
- Атомарный `claimAppeal(... 'denied')`, фиксируется `resolvedAt` (нужен для кулдауна 48 ч).
- ЛС участнику об отказе с датой, когда можно подать новую апелляцию. Embed красный с пометкой,
  кнопки неактивны, запись в `decisions`, канал-вопрос (если был) удаляется.

#### 3.3. ❓ Задать вопрос
- Если канал-вопрос уже существует — «Канал с вопросом уже существует: <#…>» (повторно не создаётся).
- Если участник вышел — «Пользователь покинул сервер.».
- Иначе создаётся приватный канал с правами для участника и ролей `roles.ststaff`,
  ID канала сохраняется в апелляции. Апелляция остаётся `pending` — решение принимается позже теми же кнопками.
- Кнопка **«Закрыть вопрос»** внутри канала доступна с уровня **staff** и выше
  (`hasButtonAccess(..., 'staff')`).

### 4. Конкурентность

- Решения по апелляции идут через атомарный `claimAppeal` (UPDATE … WHERE status='pending') —
  двойная обработка исключена, второму модератору сообщается актуальный статус.

### 5. Обзор очереди

- `/амнистии` (доступ `mod`) — список всех `pending`-апелляций: номер, упоминание, тег, время, ссылка.
  Пусто — «Непринятых апелляций нет. 🎉». Ответ эфемерный.

### 6. Что модератор видит при авто-событиях

- **left**: на сообщении апелляции появляется неактивная кнопка «Покинул сервер».
  (Авто-expire для апелляций, в отличие от анкет, не реализован — кулдаун управляет лишь
  повторной подачей со стороны участника.)

## Приложение. Карта переходов статусов

**Анкета:**
```
            ┌────────────► approved   (Принять)
            │
 (нет)──► pending ─────────► rejected   (Отклонить + причина)  ──► можно подать заново
            │  ▲
            │  │ (Задать вопрос: остаётся pending, создаётся канал-вопрос)
            │
            ├────────────► blacklisted (ЧС + причина, или /выдатьчсп) ──► далее только апелляция
            ├────────────► expired     (авто, 48 ч)     ──► можно подать заново
            └────────────► left        (вышел с сервера)──► можно подать заново
```

**Апелляция (доступна только при роли ЧС):**
```
 pending ──► amnestied  (Принять амнистию)   ──► ЧС снят, доступна верификация
        ├──► denied     (Отказать)           ──► кулдаун 48 ч, затем новая апелляция
        └──► left       (вышел с сервера)
```
